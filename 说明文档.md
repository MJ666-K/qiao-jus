# 枫桥智诉 · 检索与模型说明

本文说明系统中**向量模型**、**重排序**、**BM25** 以及**混合检索**的实现方式，便于理解问答/检索测试为何能召回相关材料。

---

## 一、向量模型（Embedding）

### 用的是什么？

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `embedding_model_id` | `text-embedding-v3` | 向量模型名称 |
| `embedding_dim` | `1024` | 向量维度 |
| `llm_api_url` | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 调用地址（OpenAI 兼容） |
| `llm_api_key` | 环境变量 `LLM_API_KEY` | 与对话大模型共用同一套 API Key |

实现位置：`knowledge-service/src/core/llm.py` 的 `embed_texts()`，通过 **LiteLLM** 调用 OpenAI 兼容接口。

### 工作原理

1. 文档入库时：对每个**子块（child chunk）** 文本调用 Embedding API，得到 1024 维向量。
2. 向量写入 **Qdrant** 向量库（collection 默认名 `knowledge`）。
3. 用户提问时：先把问题转成同样维度的向量，再在 Qdrant 里做**余弦相似度**近邻搜索（Dense 检索）。

### 可替换吗？

可以。在 `knowledge-service/.env` 中修改 `EMBEDDING_MODEL_ID`、`EMBEDDING_DIM`、`LLM_API_URL` 等，需保证维度与 Qdrant 配置一致。  
**注意**：换模型后已有向量不会自动更新，需对文档重新索引（reindex）。

## 二、重排序模型（Reranker）

### 用的是什么？

使用阿里云 DashScope **专用重排 API**，模型为 **`qwen3-rerank`**（OpenAI 兼容接口）。

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `rerank_model_id` | `qwen3-rerank` | 重排模型 |
| `rerank_api_url` | `https://dashscope.aliyuncs.com/compatible-api/v1/reranks` | 重排接口地址 |
| `rerank_instruct` | 英文检索指令（可配置） | 告诉模型按何种任务排序 |
| `llm_api_key` | 环境变量 `LLM_API_KEY` | 与 Embedding / 对话共用 |

实现位置：
- API 调用：`knowledge-service/src/core/llm.py` → `rerank_documents()`
- 封装：`knowledge-service/src/retrieve/reranker.py`

### 工作原理

1. RRF 融合后取 Top-N 候选（默认 20 条），每条带上**父块完整文本**。
2. 将 `query` + `documents[]` 发给 `/reranks` 接口。
3. 模型返回 `index` + `relevance_score`，按分数重排后取最终 Top-K。

请求示例（与线上一致）：

```bash
curl --request POST \
  --url https://dashscope.aliyuncs.com/compatible-api/v1/reranks \
  --header "Authorization: Bearer $DASHSCOPE_API_KEY" \
  --header "Content-Type: application/json" \
  --data '{
    "model": "qwen3-rerank",
    "query": "用户问题",
    "documents": ["候选片段1", "候选片段2"],
    "top_n": 10,
    "instruct": "Given a web search query, retrieve relevant passages that answer the query."
  }'
```

若 API 调用失败，会**降级**为保持 RRF 融合后的顺序，不影响主流程。

### 与系统配置的关系

「系统配置 → 文本切块规则」中的 **Rerank Top-K**（默认 20）表示：RRF 融合后先取前 20 个候选送去重排，最终再按 **Top-K 召回**（默认 10）返回给问答。

## 三、BM25 是什么？用的是什么「模型」？

### BM25 不是神经网络模型

**BM25（Best Matching 25）** 是经典的信息检索**稀疏检索算法**，基于词频统计，不需要 GPU、也不调用外部 API。

本项目使用 Python 库 **`rank-bm25`**（`BM25Okapi`），实现见 `knowledge-service/src/retrieve/bm25.py`。

### 分词方式

- 中文：按单字 + 连续英文/数字切分（正则 `[\u4e00-\u9fff]|[a-zA-Z0-9]+`）
- 纯英文：按单词切分

### 主要参数（可在系统配置中调整）

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `bm25_k1` | 1.5 | 词频饱和度，越大越重视词出现次数 |
| `bm25_b` | 0.75 | 文档长度归一化，越大长文惩罚越重 |

### BM25 检索的数据范围

检索时从 PostgreSQL 加载当前租户（及可选知识库）下的**父块（parent chunk）** 文本，在内存中构建 BM25 索引（带约 5 分钟缓存）。

> **说明**：向量检索打在**子块**上，BM25 打在**父块**上，两者粒度不同；融合后再统一取父块上下文给大模型，这是「父子块 RAG」设计的一部分。

## 四、混合检索（Hybrid Retrieval）

核心入口：`retrieve_children()`（`knowledge-service/src/retrieve/hybrid.py`）

用于：**我的助手**问答召回、**检索测试**、图谱局部查询等需要「从知识库找片段」的场景。

### 整体流程

```
用户问题
    │
    ├─► [1] Embedding 向量化
    │
    ├─► [2] 双路召回（并行）
    │       ├─ Dense：Qdrant 向量相似度（子块）
    │       └─ Sparse：BM25 关键词匹配（父块）
    │
    ├─► [3] RRF 融合（Reciprocal Rank Fusion）
    │       把两路排名合并，不依赖分数尺度
    │
    ├─► [4] 取融合 Top-N（rerank_top_k，默认 20）
    │       并拉取对应父块完整文本
    │
    ├─► [5] 重排序（qwen3-rerank API）
    │
    └─► [6] 去重 + 截断 Top-K（search_top_k，默认 10）
            返回给大模型做引用与回答
```

### 各阶段说明

#### 1. Dense 向量召回

- 问题 → `text-embedding-v3` → 1024 维向量
- 在 Qdrant 中按 `tenant_id`、`dataset_id` 等过滤后做近邻搜索
- 召回数量：`top_k × dense_top_k_multiplier`（默认 10×3=30）

#### 2. BM25 稀疏召回

- 对同一租户/知识库下的父块建 BM25 索引
- 按 query 词项匹配打分
- 召回数量：与 Dense 相同（`extend`）

#### 3. RRF 融合

**Reciprocal Rank Fusion（倒数排名融合）** 公式：

\[
\text{score}(d) = \sum \frac{1}{k + \text{rank}(d)}
\]

- `k` 由 `rrf_k` 控制（默认 60）
- 优点：Dense 分数（余弦）和 BM25 分数（词频）量纲不同，RRF 只比**排名**，融合更稳

#### 4. 父子块上下文（Parent-Child RAG）

- **子块**：体积小、语义集中，适合向量检索命中
- **父块**：上下文更完整，适合拼进 Prompt 给大模型
- 命中子块后，系统会带上对应**父块全文**（若无父块则用子块自身）

#### 5. 重排序（qwen3-rerank）

调用 DashScope `compatible-api/v1/reranks`，对 RRF 后的候选父块文本做语义相关性精排，返回 `relevance_score`。  
可通过 **系统配置 → 检索与模型** 调整 `rerank_model_id`、`rerank_instruct`（重排指令）及 `rerank_top_k`。

### 可调参数汇总（系统配置 → 检索与模型）

| 配置项 | 默认值 | 作用阶段 |
|--------|--------|----------|
| `search_top_k` | 10 | 最终返回条数 |
| `dense_top_k_multiplier` | 3 | Dense/BM25 每路召回扩展倍数 |
| `rrf_k` | 60 | RRF 融合平滑系数 |
| `rerank_top_k` | 20 | 送入重排的候选数 |
| `rerank_model_id` | `qwen3-rerank` | 重排模型 |
| `rerank_instruct` | 英文检索指令 | 重排 API instruct 字段 |
| `bm25_k1` / `bm25_b` | 1.5 / 0.75 | BM25 算法参数 |
| `chunk_parent_tokens` 等 | 1200/300/50 | **仅影响新文档切块**，不参与在线检索公式 |

检索、重排与大模型参数保存后对**下一次问答/检索**生效（API 进程内更新，检索结果有约 60 秒缓存）。  
切块参数仅影响**新上传文档**的切分方式，存量文档需重新索引。

## 五、对话大模型（补充）

与检索无关，但常与上述模型一起配置：

| 配置项 | 默认值 | 用途 |
|--------|--------|------|
| `llm_model_id` | `qwen-plus` | 助手对话、报告生成、图谱实体抽取等（`.env` 配置） |
| `llm_max_tokens` | 8000 | 单次最大输出长度 |
| `llm_stream_temperature` | 0.4 | 智能问答流式回复 |
| `llm_chat_temperature` | 0.2 | 非流式对话（检索摘要等） |
| `llm_json_temperature` | 0.1 | JSON 结构化输出（报告、图谱） |
| `llm_suggest_temperature` | 0.3 | 问答结束后的追问建议 |

前四项除 `llm_model_id` 外均可在 **系统配置 → 检索与模型** 中在线调整。  
同样通过 LiteLLM + DashScope OpenAI 兼容接口调用，API Key 与 Embedding 共用。

## 七、一句话总结

- **向量模型**：默认阿里云 DashScope **`text-embedding-v3`（1024 维）**，存 Qdrant，负责语义相似召回。  
- **BM25**：**经典词频算法**（`rank-bm25` 库），不是深度学习模型，负责关键词精确匹配。  
- **混合检索**：Dense + BM25 **双路召回 → RRF 融合 → qwen3-rerank 重排 → 父子块上下文**，兼顾语义与关键词。  
- **重排序**：使用 DashScope **`qwen3-rerank`** 专用重排 API（`compatible-api/v1/reranks`）。

# 枫桥智诉 · 思路说明

> 用一段话讲清楚：这是一个**法律智能辅助平台**，让不懂法的人也能像有个律师朋友一样，上传材料、看懂风险、追问细节。

---

## 一、要解决什么问题？

普通用户（街道调解员、中小企业、信访群众）遇到法律问题，通常面临三个难题：

1. **看不懂** —— 合同里的条款、法律条文，专业术语太多
2. **不知道怎么办** —— 发生纠纷，不知道该准备什么证据、走什么流程
3. **问不到** —— 真律师咨询贵、排队久，AI 又怕胡说

我们的思路：**让 AI 读法律 + 读用户材料，给出"看得懂、有依据、可追问"的答案。**

---

## 二、核心思路：三步走

所有功能都遵循同一个套路：

```
   ① 上传材料  →  ② AI 生成报告  →  ③ 智能问答追问
       │              │                  │
   合同/纠纷/        结构化风险点        基于报告继续聊
   证据材料          + 法条引用          想问什么问什么
```

**为什么是这三步？**

| 步骤 | 解决什么 | 价值 |
|------|---------|------|
| 上传 | 系统拿到用户的"具体材料" | 不再是空泛问答，有针对性 |
| 报告 | AI 把复杂材料**结构化**成结论 | 用户一眼看清重点、风险 |
| 问答 | 用户能针对报告**追问** | 解决"看不懂哪里"的最后一步 |

> 举个例子：上传一份劳动合同 → 系统生成《合同审查报告》指出 3 处风险 → 用户追问"第 2 条不改会怎样？"→ AI 结合报告和平台类案回答。

---

## 三、两个知识库（关键设计）

系统里有两套数据，这是理解整个系统的核心：

```
┌───────────────────────────────────────┐
│   平台公共知识库（全员共享，管理员维护）     │
│   法规 · 类案 · 合规条款 · 知识图谱        │
└──────────────────┬────────────────────┘
                   │  AI 联合检索
┌──────────────────┴────────────────────┐
│   用户私有库（按用户隔离，本人可见）        │
│   合同 · 纠纷材料 · 生成的报告             │
└───────────────────────────────────────┘
```

**为什么分两个库？**

- **平台库**：大家共享的"法律百科"，管理员统一维护，保证权威
- **用户库**：每个人的材料是私密的，互相看不到

AI 回答问题时，会**同时查两个库**：用户的具体情况 + 平台的法律依据，所以答案既"懂你"又"有依据"。

## 四、三项核心技术

### 1. RAG 检索（让 AI 看得懂法律）

AI 本身不懂法律细节，怎么办？**先从库里找相关法条/案例，再让 AI 读着回答。**

```
用户提问 → 找相关法条/案例（混合检索）→ AI 依据找到的内容回答 → 带引用
```

**混合检索 = 两路并行，取长补短：**
- **向量检索**：找"意思相近"的内容（懂语义）
- **关键词检索**：找"字面对得上"的内容（抓精确，比如法条编号）
- 两路结果融合 → 既准又全

### 2. 知识图谱（让 AI 懂关系）

法律里很多关系是固定的，比如：

```
法条"劳动合同法第39条" ——适用于——→ 案由"违法解除劳动合同"
                                    │
                                  需要
                                    ▼
                              证据"解除通知书、工资流水"
```

这些关系画成一张**图**存进 Neo4j。当用户问"还需要什么证据？"时，AI 顺着图就能推导出来——不用猜，**关系就在那**。

**什么时候用图谱 vs RAG？**

| 问题 | 用什么 | 原因 |
|------|--------|------|
| "类似案例怎么判？" | RAG | 要读判决书原文 |
| "需要准备哪些证据？" | 图谱 | 固定关系，顺着图查 |
| "第39条说什么？" | RAG | 要法条原文 |
| "哪些法条管这事？" | 图谱 | 法条↔案由关系链 |

### 3. Agent 智能体（让 AI 会选工具）

不让 AI 直接硬答，而是给它一排**工具**，让它自己判断该用哪个：

```
           ┌─ 查法规 ─┐
用户问题 → ├─ 查类案 ─┤ → 综合判断 → 回答
           ├─ 查图谱 ─┤
           ├─ 查报告 ─┤
           └─ 查用户材料 ─┘
```

比如问"我合同第2条风险严重吗？"，Agent 会：①读你的合同 → ②查相关法条 → ③查类似判例 → 综合作答。**像律师的工作流程。**

## 五、五大功能模块

五个模块面向不同场景，但**底层套路完全一样**（都是"三步走"）：

| 模块 | 给谁用 | 上传什么 | 生成什么 |
|------|--------|---------|---------|
| **M1 纠纷研判** | 街道/信访 | 纠纷描述 | 纠纷研判报告（案由+证据+建议） |
| **M2 合同审查** | 中小企业 | 合同 PDF | 合同审查报告（风险点+法条） |
| **M3 用工排查** | 中小企业 | 用工问卷 | 用工风险报告 |
| **M4 文书生成** | 群众/律师 | 案由+事实 | 起诉状/答辩状草稿 |
| **M5 证据指引** | 街道/群众 | 选择案由 | 证据清单+流程指引 |

> **关键洞察**：五个模块复用同一套底层（上传 → 报告 → 问答），先做通一个，其他照搬。这是控制开发成本的核心。

---

## 六、智能问答：能"接着聊"

问答不是一问一答就完，而是**像聊天一样多轮追问**：

- **长连接**：一条 WebSocket 通道，反复问
- **带记忆**：记得前面聊过什么
- **流式输出**：答案一个字一个字冒出来，不用干等
- **引用溯源**：每条答案都标明出自哪份文件哪一段，点击跳原文

**两种问答模式：**

| 模式 | 怎么触发 | AI 查什么 |
|------|---------|----------|
| **通用问答** | 直接开聊 | 平台法规+类案+图谱 |
| **报告问答** | 从报告页"基于此报告问答" | 你的报告+你的材料+平台库（三路） |

## 七、整体架构一图流

```
        用户浏览器（Vue 前端）
              │
        ─────────────
              │
       后端服务（FastAPI）
              │
    ┌─────────┼─────────┐
    │         │         │
  Skill层  Pipeline层  Retrieve层
  (功能模块) (文档处理)  (检索)
    │         │         │
    └─────────┼─────────┘
              │
   ┌──────────┼──────────┐
   │          │          │
PostgreSQL  Qdrant     Neo4j
(业务数据+   (向量      (知识
 关键词)     检索)       图谱)
```

**三库分工：**
- **PostgreSQL**：存文档、用户、对话这些"业务数据"
- **Qdrant**：存"向量"（把文字变成数字），用来算语义相似度
- **Neo4j**：存"关系图"，用来做多跳推理

---

## 八、总结：思路的三层价值

1. **对用户**：把"找律师"这件事，从**贵+慢+怕被骗**，变成**随时问+免费+有依据**
2. **对系统**：五大模块共用一套底层，**做一个就能复制五个**，性价比高
3. **对答案质量**：RAG（找原文）+ 图谱（查关系）+ Agent（选工具）三件套，答案**既准又能溯源**，不是 AI 瞎编

> 一句话总结：**让 AI 依法说话，让普通人看得懂法。**
